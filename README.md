# Лабораторная работа: Инцидент в АИС — обнаружение, расследование и восстановление

**Цель работы:** Наглядно изучить атаку на хранилище данных АИС через незащищённый внутренний сетевой интерфейс, провести расследование инцидента по следам в базе данных и логах, самостоятельно устранить последствия атаки и предложить меры защиты.

---

## Описание стенда

Вам предоставляется изолированная учебная среда из четырёх контейнеров:

| Контейнер | Роль | Доступ |
|---|---|---|
| `ais-web-sN` | Веб-приложение (кликер с таблицей лидеров) | Браузер, порт `800N` |
| `ais-redis-sN` | База данных Redis (in-memory) | Только внутренняя сеть |
| `ais-ssh-sN` | Ваш «рабочий» контейнер | SSH, порт `220N` |
| `ais-attacker-sN` | Симулятор атаки *(скрытый)* | — |

Все контейнеры подключены к **одной внутренней Docker-сети** (`ais-net-N`). Redis работает **без пароля** — это и есть уязвимость.

---

## Ход работы

### Шаг 1. Нормальная эксплуатация

1. Откройте веб-страницу вашего приложения (порт выдаёт преподаватель).
2. Ознакомьтесь с интерфейсом: слева — ваш счётчик кликов, справа — **таблица лидеров** с несколькими игроками.
3. Сделайте 10–20 кликов, убедитесь, что ваш результат (`Student_N`) отображается в таблице.
4. Обновите страницу — данные должны сохраниться (сервер записал их в Redis).

> **Вопрос для размышления:** почему счёт сохраняется после обновления страницы? Какой компонент за это отвечает?

---

### Шаг 2. Наблюдение за инцидентом

**Не закрывайте браузер.** Продолжайте наблюдать за таблицей лидеров — она обновляется автоматически каждые 3 секунды.

Примерно через **1–2 минуты после запуска** среды в таблице лидеров появится новая запись:

```
#1   h4ck3r_b0t     999 999
```

Одновременно на странице отобразится красное предупреждение об аномалии.

> **Зафиксируйте момент появления записи** (скриншот). Это ключевое событие инцидента.

---

### Шаг 3. Расследование — анализ базы данных

Подключитесь к вашему SSH-контейнеру:

```bash
ssh student@<IP-адрес> -p 7000+ваш номер
# пароль: studentN
```

Установите клиент Redis:

```bash
sudo apk add --no-cache redis
```

Подключитесь к базе данных и изучите её состояние:

```bash
redis-cli -h redis
```

Выполните следующие команды и **запишите вывод каждой** — они войдут в отчёт:

```
# 1. Все записи в таблице лидеров (от большего к меньшему):
ZRANGE leaderboard 0 -1 WITHSCORES REV

# 2. Сколько записей всего:
ZCARD leaderboard

# 3. Место и счёт подозрительного игрока:
ZRANK leaderboard h4ck3r_b0t
ZSCORE leaderboard h4ck3r_b0t

# 4. Посмотреть все ключи в базе:
KEYS *
```

> **Вопрос для отчёта:** является ли значение `999999` результатом легитимных кликов? Каким способом оно могло быть установлено?

Выйдите из redis-cli (`exit` или `Ctrl+D`), но **оставайтесь в SSH**.

---

### Шаг 4. Расследование — анализ логов

Представим, что вы нашли логи иувидели строку кода:

```
... [N] command=ZADD, args=leaderboard 999999 h4ck3r_b0t from <IP-адрес контейнера>
```

> **Зафиксируйте IP-адрес источника команды.** Он принадлежит контейнеру `ais-attacker-sN`.
>
> **Вопрос:** что позволило атакующему контейнеру обратиться к Redis напрямую? Почему сервер приложения (`ais-web-sN`) не обнаружил атаку?

---

### Шаг 5. Устранение последствий (Rollback)

Теперь вам предстоит **самостоятельно** исправить скомпрометированные данные, не перезапуская всю систему.

Из SSH-контейнера (там у вас уже установлен redis-cli):

**Вариант А — точечное удаление вредоносной записи:**

```bash
redis-cli -h redis ZREM leaderboard h4ck3r_b0t
```

Проверьте результат в браузере — запись `h4ck3r_b0t` должна исчезнуть из таблицы лидеров.

**Вариант Б — полный сброс базы данных** *(если данные считаются полностью скомпрометированными)*:

```bash
redis-cli -h redis FLUSHALL
```

> ⚠ FLUSHALL удаляет **все** данные, включая ваши легитимные очки. Сервер пересоздаст начальное состояние при следующем обращении к API.

Убедитесь в браузере, что таблица лидеров вернулась в корректное состояние.


## Требования к отчёту

Отчёт должен содержать:

### 1. Описание атаки
- Какой вектор атаки был использован?
- Почему Redis оказался доступен напрямую?
- Что именно было изменено в базе данных и какой командой?

### 2. Доказательная база (скриншоты)
- Таблица лидеров **до** инцидента (нормальное состояние).
- Таблица лидеров **в момент** появления `h4ck3r_b0t`.
- Вывод команды `ZRANGE leaderboard 0 -1 WITHSCORES REV` из redis-cli.
- Фрагмент логов Redis с командой `ZADD`.
- Таблица лидеров **после** устранения последствий.

### 3. Анализ уязвимости
Ответьте на вопросы:
- Относится ли данная атака к типу «инсайдерская угроза», «атака на цепочку поставок» или «атака через внутреннюю сеть» (lateral movement)? Аргументируйте.
- Нарушение каких свойств информационной безопасности произошло: **конфиденциальность**, **целостность**, **доступность**? Для каждого — обоснование.

### 4. Меры по предотвращению

Предложите не менее **трёх конкретных технических мер**, которые устранили бы уязвимость. Для каждой укажите:
- что именно нужно изменить в конфигурации/архитектуре,
- какую угрозу это устраняет.

*Подсказка: обратите внимание на аутентификацию Redis, сетевую изоляцию контейнеров и разграничение прав доступа.*
